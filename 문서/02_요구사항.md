## 2.1 기능 요구사항 (FR)

### FR-1. 테마 생성
| ID | 요구사항 | 우선순위 |
|---|---|---|
| FR-1.1 | 기준 색(Seed Color)을 지정할 수 있어야 한다 | 필수 |
| FR-1.2 | 배색 방식(Harmony)을 선택해 팔레트를 자동 생성해야 한다 | 필수 |
| FR-1.3 | 이미지를 첨부하면 대표 색을 추출해 테마를 구성해야 한다 | 필수 |
| FR-1.4 | 추출 색을 역할별(배경/텍스트/강조)로 매핑할 수 있어야 한다 | 권장 |
| FR-1.5 | 랜덤 팔레트를 생성하고 사용된 색상 모음을 표시해야 한다 | 권장 |
| FR-1.6 | 올해의 트렌드 컬러(예: Pantone 2026 Cloud Dancer)를 추천 제공해야 한다 | 권장 |

### FR-2. 접근성 검증
| ID | 요구사항 | 우선순위 |
|---|---|---|
| FR-2.1 | 역할쌍(배경↔텍스트, 배경↔포인트)별 WCAG 대비비를 자동 계산해야 한다 | 필수 |
| FR-2.2 | AA/AAA 통과 여부를 표로 제시해야 한다 | 필수 |
| FR-2.3 | AAA를 통과한 색만 콘텐츠 색상으로 사용되도록 필터링해야 한다 | 필수 |
| FR-2.4 | 미달 항목을 자동 보정하고, 무엇이 어떻게 바뀌었는지 피드백해야 한다 | 필수 |
| FR-2.5 | 사용자가 직접 지정한 색도 대비비 검증 대상에 포함해야 한다 | 필수 |
| FR-2.6 | 검증표는 화면을 막지 않는(비차단) 형태로 실시간 반영을 확인할 수 있어야 한다 | 필수 |

### FR-3. 테마 관리 · 비교

## 2.2 비기능 요구사항 (NFR)

| ID | 분류 | 요구사항 |
|---|---|---|
| NFR-1 | 사용성 | 비개발자가 학습 없이 사용 가능해야 하며, 주요 기능에 툴팁·설명을 제공한다 |
| NFR-2 | 사용성 | 과도한 그림자·효과를 지양하여 시각 피로를 최소화한다 |
| NFR-3 | 접근성 | 폰트 최소 크기 12px, WCAG AAA 대비 기준을 준수한다 |
| NFR-4 | 반응형 | 우측 콘텐츠는 화면 폭에 유연하게 반응하고, 전체 높이 100vh에 맞춘다 |
| NFR-5 | 성능 | 외부 프레임워크 없이 단일 HTML로 구동하여 배포·이식이 간단해야 한다 |
| NFR-6 | 이식성 | 별도 서버 없이 정적 호스팅(GitHub Pages)만으로 동작해야 한다 |
| NFR-7 | 일관성 | 좌측 미세조정은 우측 콘텐츠에만 반영하고 좌측 메뉴 색은 고정한다 |
| NFR-8 | 국제화/폰트 | 한글 최적화 폰트(Pretendard)를 CDN으로 적용한다 |

## 2.3 제약 사항 (Constraints)

- 단일 `index.html` 파일 구조 유지 (Vanilla JS, 외부 빌드 도구 미사용)
- 백엔드 없음 — 상태는 브라우저(localStorage) 및 공유 URL로만 관리
- 날씨 등 외부 데이터는 클라이언트에서 직접 호출
- 관제 지표는 시뮬레이션 데이터 기반 (실데이터 연동은 범위 외)

---

## 2.4 요구사항 이력 관점

요구사항은 STUDIO 세션 01~12에 걸쳐 반복적으로 상세화·개선되었으며, 특히 다음 항목이 사용자 피드백으로 강화되었다.
- 접근성 자동 보정(단순 표시 → 자동 교정 + 변경 피드백)
- 시각 체계 이원화(단일 → Flat / Enterprise)
- 미세조정 반영 범위 제어(전체 반영 → 우측 콘텐츠 한정)
- 라이트 모드 완성도(배경 밝기·팝업/모달·버튼 가시성)
